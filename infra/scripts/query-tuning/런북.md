# 쿼리 튜닝 하네스 — 실행 런북

콜드 스타트부터 대시보드 확인까지 **순서대로 따라 실행**하는 절차서. "무엇을·왜"는 `README.md`,
설계 원리는 `../../docs/query-tuning/STEP1~4` 참조. 이 문서는 **"이 순서대로 치면 된다"**만 담는다.

## 컨벤션 (고정)

| 레벨 | 컨테이너 | 호스트 포트 | 규모(problem) |
|---|---|--:|--:|
| x1 | `local-mysql-x1` | 3307 | 71,244 (원본) |
| x10 | `local-mysql-x10` | 3308 | ~726k |
| x100 | `local-mysql-x100` | 3309 | ~7.28M (분석 대상, 대시보드가 읽음) |

- 전부 `root`/`password`, DB `qaskerdb`. 모두 `127.0.0.1` 바인딩(공개 금지) + `local_local-monitoring` 네트워크.
- 명령은 모두 `api/` 에서 실행한다.

---

## 0. 사전 준비 (최초 1회)

```bash
# PLG 스택(Grafana·Prometheus·Loki + local_local-monitoring 네트워크) 기동
cd ../plg-stack && docker compose up -d && cd ../api
docker network ls | grep local_local-monitoring   # 네트워크 존재 확인(provision-level 이 여기 붙는다)

# jq 설치 확인(로드젠·요약에 사용)
command -v jq || brew install jq

# OCI CLI + BACKUP_READER 프로파일(2단계 restore-x1 --latest 가 마스킹본을 내려받는다)
command -v oci && grep -q BACKUP_READER ~/.oci/config
```

- **Grafana**: http://localhost:3000 — 로그인 `admin` / `inha-cc-01`.
  (기본값이 안 먹으면: `docker exec local-grafana grafana cli admin reset-admin-password inha-cc-01`)
- MySQL 데이터소스(`qasker-mysql-local` → `host.docker.internal:3309`)는 **프로비저닝 자동 등록**이라 손댈 것 없음.

---

## 1. 컨테이너 생성 (x1/x10/x100)

빈 컨테이너를 prod-matched config(버퍼풀 4G·O_DIRECT 등)로 만든다. 데이터는 2·3단계가 채운다.

```bash
bash infra/scripts/query-tuning/provision-level.sh local-mysql-x1   3307
bash infra/scripts/query-tuning/provision-level.sh local-mysql-x10  3308
bash infra/scripts/query-tuning/provision-level.sh local-mysql-x100 3309
# 기존 볼륨 보존 재생성: 3번째 인자로 볼륨명 (docker volume ls / inspect 로 확인)
```

> ⚠️ 3레벨 모두 버퍼풀 4G라 동시 상주 시 RAM 압박. 스윕은 한 번에 한 레벨만 띄운다(run-all 이 자동 관리).

---

## 2. x1 채움 (OCI 마스킹본 → `local-mysql-x1` + 정합성 검증)

**책임 분리**: 마스킹 덤프 **생성은 plg-stack**(masked-export), **다운로드·복원·검증은 api**.

```bash
# (1) 다운로드 — OCI 최신 마스킹본 → 파일 (download-masked 가 sha256 검증, 경로 출력)
DUMP=$(bash infra/scripts/query-tuning/download-masked.sh --latest)
# (2) 복원 — 그 파일을 x1 에 복원 + row·FK 정합 검증
bash infra/scripts/query-tuning/restore-x1.sh "$DUMP"
# 이미 로컬에 덤프가 있으면 (1) 생략: restore-x1.sh /path/masked.sql.gz
```

> 두 스크립트는 **독립**(다운로드 / 복원 분리). OCI 접근은 `BACKUP_READER` 프로파일(`~/.oci/config`). 마스킹본이라 원본 PII 미도달.

---

## 3. 배수 시딩 (초기화+복원+복제 한 방)

`seed-scale.sh` 가 **대상 DROP → x1 덤프 복원 → 배수 복제 → 검증**을 한 커맨드로 한다.
**재실행해도 매번 fresh** x\<배수\>(누적·중복키 없음). 배수만 바꾸면 어떤 스케일도 재현.

```bash
bash infra/scripts/query-tuning/seed-scale.sh local-mysql-x10  10
bash infra/scripts/query-tuning/seed-scale.sh local-mysql-x100 100
# x1 소스가 다른 컨테이너면 3번째 인자로 지정 (기본 local-mysql-x1, 정지 상태면 자동 기동·원복)
```

각 실행 끝에:
- **충분성 가드**(`check-x1-scale.sh`)가 자동 실행 — x1×100 이 **10K 미만인 도메인 테이블을 경고**(진행은 안 막음).
  새 도메인 테이블이 생겨도 information_schema 자동 발견으로 검사된다.
- **총량 검증표**(원본 × 배수) 출력.

---

## 4. 스케일 스윕 + 요청 귀속 → 대시보드 §①②③

앱을 각 레벨 DB에 붙여 부하를 태우고 요청을 SQL 주석으로 귀속(`trace_snapshot`)한다.

```bash
ROUNDS=50 bash infra/scripts/query-tuning/run-all.sh          # 전 레벨 순차
# 또는 특정 레벨만
bash infra/scripts/query-tuning/run-all.sh x100
```

- `run-all` 이 스윕 전 bootJar 1회 빌드 → 레벨별 `run-level` 호출. 중간 레벨(x1·x10)은 끝나면 정지(RAM 반환), **x100(3309)만 상주**(대시보드가 읽음).
- `run-level` 은 **실데이터·볼륨을 안 건드린다** — `trace_snapshot`·perf_schema·slow_log 만 이번 실행분으로 갱신.
- 개별 호출:
  ```bash
  bash infra/scripts/query-tuning/run-level.sh 3307 local-mysql-x1   x1
  bash infra/scripts/query-tuning/run-level.sh 3308 local-mysql-x10  x10
  bash infra/scripts/query-tuning/run-level.sh 3309 local-mysql-x100 x100
  ```

## 5. 대시보드 확인

- **[쿼리튜닝] 통합 진단** (uid `q-asker-unified-diagnosis`) — http://localhost:3000/d/q-asker-unified-diagnosis
- 좌상단 `레포지토리` 변수로 대상 레포 선택.
  - **§①**(Prometheus): 스케일별 레포·메서드 지연 — 크기 의존 O(n) 탐지.
  - **§②③**(MySQL `trace_snapshot`): 엔드포인트별 요청당 쿼리수(N+1)·스캔행수·대표 SQL 시퀀스.
  - **§④**(Prometheus): 레포별 읽기/쓰기 비중. **부록**: slow_log.
- §②③이 "no data" 면 §4 스윕이 안 돌았거나 MySQL 데이터소스가 x100(3309)를 못 봄 → 아래 트러블슈팅.

---

## 재시딩 / 배수 변경

배수를 바꾸거나 데이터를 초기화하려면 **3단계 seed-scale 를 다시 실행**하면 된다(초기화 포함이라 그대로 재실행 안전).

```bash
bash infra/scripts/query-tuning/seed-scale.sh local-mysql-x100 200   # x200 으로 재현
```

> "x100 을 다시 돌리면 x200 이 되나?" → **아니다.** seed-scale 은 매번 DROP+복원부터 하므로 항상 fresh x\<배수\>다.
> 재실행 누적·중복키 걱정 없음.

## 트러블슈팅

| 증상 | 원인 / 조치 |
|---|---|
| `run-level`: "컨테이너 없음" | provision(2) + seed(3) 를 먼저. run-level 은 컨테이너를 자동 생성하지 않음(실수 방지). |
| `seed-scale`: "x1 소스 없음" | 1단계(provision) + 2단계(`restore-x1.sh`) 로 `local-mysql-x1` 먼저 채움. |
| `seed-scale`: "x1 을 대상으로 줄 수 없음" | 대상에 `local-mysql-x1` 을 넣음 — 원본 보호로 거부. x10/x100 을 대상으로. |
| 대시보드 §②③ no data | (a) 해당 레벨 `run-level` 이 안 돌아 `trace_snapshot` 없음, (b) x100 컨테이너 정지, (c) 데이터소스 헬스: `curl -s -u admin:inha-cc-01 localhost:3000/api/datasources/uid/bfrwvbnu3162ob/health`. |
| Grafana 로그인 실패 | `docker exec local-grafana grafana cli admin reset-admin-password inha-cc-01`. |
| stale `local-mysqld-exporter-*` | 옛 이름 익스포터 잔재 — `run-level` 이 재생성하거나 `docker rm -f` 로 정리. |
| 충분성 경고(10K 미만) | 정보성 경고(진행 무관). 해당 테이블 쿼리를 스케일에서 봐야 하면 x1 베이스라인 보강 검토. |
| RAM 압박 | 레벨 동시 상주 금지. run-all 이 x1·x10 을 자동 정지. 수동이면 `docker stop`. |

## 정리 (측정 종료 후)

```bash
docker stop local-mysql-x1 local-mysql-x10          # x100 은 대시보드용으로 유지 가능
# 완전 제거(볼륨은 남음): docker rm -f local-mysql-x1 local-mysql-x10 local-mysql-x100
# 볼륨까지: docker volume ls | grep <해시> 확인 후 docker volume rm
```
